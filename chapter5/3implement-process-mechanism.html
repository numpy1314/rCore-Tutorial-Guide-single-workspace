进程管理机制的设计实现¶

本节导读¶

本节将介绍如何基于上一节设计的 Process、ProcManager 和 Processor 数据结构来实现完整的进程管理功能：

  • 初始进程创建：构建系统启动后的第一个用户进程 initproc。

  • 进程调度机制：实现 sys_yield 和时间片轮转，利用 ForeignContext 进行特权级切换。

  • 进程生成机制：深入解析 sys_fork 和 sys_exec 的实现细节，特别是地址空间的复制与替换。

  • 字符输入机制：实现 sys_read 系统调用，处理标准输入。

  • 进程资源回收机制：当进程调用 sys_exit 退出后，父进程如何通过 sys_waitpid 回收僵尸进程资源。

初始进程的创建¶

内核初始化完毕之后，即会调用 task 子模块提供的 add_initproc 函数来将初始进程 initproc 加入任务管理器。在这之前，我们需要通过 lazy_static 初始化一个全局的初始进程实例。

 1// tg-ch5/src/main.rs
 2
 3use spin::Lazy;
 4use xmas_elf::ElfFile;
 5
 6/// 初始进程 initproc
 7let initproc_data = APPS.get("initproc").unwrap();
 8if let Some(process) = Process::from_elf(ElfFile::new(initproc_data).unwrap()) {
 9    PROCESSOR.get_mut().set_manager(ProcManager::new());
10    PROCESSOR.get_mut().add(process.pid, process, ProcId::from_usize(usize::MAX));
11}
12
13/// 将初始进程加入调度队列
14pub fn add_initproc() {
15    // 由于 Process 不支持 Clone，我们需要创建一个新副本或移动它。
16    // 在实际实现中，我们通常会在 main 函数中直接获取并插入，
17    // 或者通过 fork 的方式（如果支持内核线程）。
18    // 这里的逻辑稍微调整为直接在 main 中初始化：
19
20    /* 在 rust_main 中 */
21    let initproc_data = APPS.get("ch5b_initproc").unwrap();
22    let initproc = Process::from_elf(ElfFile::new(initproc_data).unwrap()).unwrap();
23    let pid = initproc.pid;
24    PROCESSOR.get_mut().add(process.pid, process, ProcId::from_usize(usize::MAX));
25    //设置了父进程ID为 usize::MAX，表示这是初始进程（没有父进程）
26}

我们调用 Process::from_elf 来创建一个进程控制块。这个函数封装了从 ELF 文件构建进程所需的复杂逻辑：

 1// tg-ch5/src/process.rs
 2
 3impl Process {
 4    pub fn from_elf(elf: ElfFile) -> Option<Self> {
 5        // 1. 解析 ELF Header 获取入口点
 6        let entry = match elf.header.pt2 { ... };
 7
 8        // 2. 创建地址空间并映射段
 9        let mut address_space = AddressSpace::new();
10        // ... 遍历 program headers 进行映射 ...
11
12        // 3. 分配用户栈
13        // 在地址空间的高地址分配一个用户栈，并设置栈指针
14        let stack = unsafe { alloc_zeroed(...) };
15        address_space.map_extern(...);
16
17        // 4. 映射异界传送门 (Portal)
18        // 这是关键步骤：用户进程必须映射跳板页才能进行系统调用
19        crate::process::map_portal(&address_space);
20
21        // 5. 初始化上下文
22        let mut context = LocalContext::user(entry);
23        // 设置 satp：模式为 SV39，根页表物理页号
24        let satp = (8 << 60) | address_space.root_ppn().val();
25        // 设置用户栈指针
26        *context.sp_mut() = 1 << 38; // 虚地址
27
28        Some(Self {
29            pid: ProcId::new(), // 分配 PID
30            context: ForeignContext { context, satp },
31            address_space,
32            heap_bottom: ...,
33            program_brk: ...,
34        })
35    }
36}

关键点在于第 17 行的 map_portal。在 SV39 分页模式下，当发生 Trap 时，硬件会跳转到 stvec 寄存器指向的地址。由于我们需要在不切换页表的情况下执行一段内核代码（保存用户寄存器），这段代码（Trampoline）必须同时存在于内核地址空间和所有用户地址空间的相同虚拟地址处（通常是最高页）。

进程调度机制¶

当进程主动调用 sys_yield 交出 CPU，或时间片耗尽时，内核需要进行进程切换。

 1// tg-ch5/src/main.rs (主循环逻辑)
 2
 3extern "C" fn rust_main() -> ! {
 4    // ... 初始化 ...
 5
 6    loop {
 7        // 1. 从管理器中取出一个任务
 8        let processor: *mut PManager<Process, ProcManager> = PROCESSOR.get_mut() as *mut _;
 9        if let Some(task) = unsafe { (*processor).find_next(){}     // 注意 task 是 Process 的可变引用
10
11            // 2. 执行任务 (切换到用户态)
12            // ForeignContext::execute 会阻塞直到 Trap 发生
13            unsafe { task.context.execute() };
14
15            // 3. 处理 Trap
16            use scause::{Exception, Trap};
17            match scause::read().cause() {
18                // 系统调用处理
19                Trap::Exception(Exception::UserEnvCall) => {
20                    use SyscallResult::*;
21                    // handle_syscall 返回一个状态，指示调度行为
22                    match tg_syscall::handle(Caller { entity: 0, flow: 0 }, id, args) {
23                        Ret::Done(ret) => match id {
24                            Id::EXIT => unsafe { (*processor).make_current_exited(ret) },  // 退出
25                            _ => {
26                                *ctx.a_mut(0) = ret as _;  // 设置返回值
27                                unsafe { (*processor).make_current_suspend() };  // 挂起
28                            }
29                        },
30                        Ret::Unsupported(_) => {
31                            unsafe { (*processor).make_current_exited(-2) };  // 不支持的系统调用
32                        }
33                    }
34                }
35                // 异常处理：杀死进程
36                trap => {
37                    unsafe { (*processor).make_current_exited(-3) };
38                }
39            }
40        } else {
41            // 队列为空，等待中断或关机
42            // ...
43        }
44    }
45}

与之前的版本不同，这里没有显式的 suspend_current_and_run_next 函数。调度逻辑被内嵌在内核的主循环中： 1. 取任务：fetch() 从队列头部拿 任务结构体的可变引用。 2. 执行：task.context.execute() 本质上就是一次上下文切换。 3. 放回/销毁：根据 Trap 原因，决定是将 task 重新 添加 到队尾，还是 退出 销毁。

这种基于主循环的调度结构更加扁平，避免了深层函数调用带来的栈开销，也更符合“单内核栈”的模型。

进程的生成机制¶

fork 系统调用的实现¶

fork 的核心是创建一个现有进程的精确副本。这涉及到地址空间的深拷贝。

 1// tg-ch5/src/process.rs
 2
 3impl Process {
 4    pub fn fork(&mut self) -> Option<Process> {
 5        // 1. 分配新 PID
 6        let pid = ProcId::new();
 7
 8        // 2. 复制地址空间
 9        // AddressSpace::cloneself 是 tg-kernel-vm 提供的接口
10        // 它会遍历源地址空间的所有映射区域，并在目标地址空间分配新的物理页，
11        // 然后将数据从源物理页拷贝到新物理页。
12        let parent_addr_space = &self.address_space;
13        let mut address_space: AddressSpace<Sv39, Sv39Manager> = AddressSpace::new();
14        parent_addr_space.cloneself(&mut address_space);
15
16        // 3. 映射异界传送门
17        // 新的地址空间必须包含 Portal，否则无法处理 Trap
18        crate::process::map_portal(&address_space);
19
20        // 4. 复制上下文
21        // 此时 context 中保存的是父进程 Trap 时的状态
22        let context = self.context.context.clone();
23
24        // 5. 构造新 satp
25        // 使用新页表的根物理页号
26        let satp = (8 << 60) | address_space.root_ppn().val();
27        let foreign_ctx = ForeignContext { context, satp };
28
29        // 6. 返回新进程实例
30        Some(Self {
31            pid,
32            context: foreign_ctx,
33            address_space,
34            heap_bottom: self.heap_bottom,
35            program_brk: self.program_brk,
36        })
37    }
38}

在 sys_fork 系统调用接口层，我们需要处理返回值的差异：

 1// tg-ch5/src/main.rs (handle_syscall)
 2
 3fn fork(&self, _caller: Caller) -> isize {
 4    let processor: *mut PManager<ProcStruct, ProcManager> = PROCESSOR.get_mut() as *mut _;
 5    let current = unsafe { (*processor).current().unwrap() };
 6    let parent_pid = current.pid; // 先保存父进程 pid
 7    let mut child_proc = current.fork().unwrap();
 8    let pid = child_proc.pid;
 9    let context = &mut child_proc.context.context;
10    *context.a_mut(0) = 0 as _;  // 子进程返回0
11    unsafe { (*processor).add(pid, child_proc, parent_pid) };
12    pid.get_usize() as isize  // 父进程返回子进程PID
13}

exec 系统调用的实现¶

exec 用于替换当前进程的映像。得益于 Rust 的所有权机制，我们可以直接用新创建的资源“覆盖”旧资源，旧资源会自动 Drop 并回收。

 1// tg-ch5/src/process.rs
 2
 3impl Process {
 4    pub fn exec(&mut self, elf: ElfFile) {
 5        // 1. 使用新 ELF 创建一个临时进程实例
 6        // 这会分配新的页表、物理页等资源
 7        let proc = Process::from_elf(elf).unwrap();
 8
 9        // 2. 资源替换
10        // 将当前进程的组件替换为新进程的组件
11        // 原有的 address_space 被覆盖后，会自动调用 Drop 回收旧页表和物理页
12        self.address_space = proc.address_space;
13        self.context = proc.context;
14        self.heap_bottom = proc.heap_bottom;
15        self.program_brk = proc.program_brk;
16
17        // 注意：self.pid 保持不变，因此进程 ID 不变
18    }
19}

在系统调用层：

// tg-ch5/src/main.rs

// 在 SyscallContext::exec 方法中（第405-425行）
fn exec(&self, _caller: Caller, path: usize, count: usize) -> isize {
    const READABLE: VmFlags<Sv39> = build_flags("RV");
    let current = PROCESSOR.get_mut().current().unwrap();

    // 使用函数式编程链式处理
    current
        .address_space
        // 1. 地址翻译：将用户空间指针转换为内核可访问的指针
        .translate::<u8>(VAddr::new(path), READABLE)
        // 2. 字符串解析：从原始字节创建UTF-8字符串
        .map(|ptr| unsafe {
            core::str::from_utf8_unchecked(core::slice::from_raw_parts(ptr.as_ptr(), count))
        })
        // 3. 应用查找：在预加载的应用映射表中查找
        .and_then(|name| APPS.get(name))
        // 4. ELF验证：解析ELF文件格式
        .and_then(|input| ElfFile::new(input).ok())
        // 5. 结果处理：成功执行替换或返回错误
        .map_or_else(
            || {
                // 错误分支：应用不存在或格式无效
                log::error!("unknown app, select one in the list: ");
                APPS.keys().for_each(|app| println!("{app}"));
                println!();
                -1  // 返回错误码
            },
            |data| {
                // 成功分支：执行进程映像替换
                current.exec(data);
                0   // 返回成功码
            },
        )
}

sys_read 获取输入¶

在本章中，我们初步引入了文件描述符的概念，尽管完整的文件系统尚未实现。

 1// tg-ch5/src/main.rs
 2
 3// 在 SyscallContext::read 方法中
 4fn read(&self, _caller: Caller, fd: usize, buf: usize, count: usize) -> isize {
 5    if fd == STDIN {
 6        // 1. 安全地翻译用户缓冲区地址
 7        const WRITEABLE: VmFlags<Sv39> = build_flags("W_V");
 8        if let Some(mut ptr) = PROCESSOR.get_mut().current().unwrap()
 9            .address_space.translate::<u8>(VAddr::new(buf), WRITEABLE)
10        {
11            // 2. 阻塞式读取count个字符
12            let mut ptr = unsafe { ptr.as_mut() } as *mut u8;
13            for _ in 0..count {
14                let c = tg_sbi::console_getchar() as u8;  // 阻塞等待
15                unsafe {
16                    *ptr = c;
17                    ptr = ptr.add(1);
18                }
19            }
20            count as _  // 返回实际读取的字节数
21        } else {
22            // 3. 地址翻译失败
23            log::error!("ptr not writeable");
24            -1
25        }
26    } else {
27        // 4. 不支持的文件描述符
28        log::error!("unsupported fd: {fd}");
29        -1
30    }
31}

注意：在真实的实现中，如果 console_getchar 返回 0，我们应当让进程进入等待状态，而不是忙轮询。但为了简化 ch5 的实现，通常策略是： 1. 用户库 wait 封装了循环。 2. 内核发现无输入时返回特定错误码或 Yield，由用户库决定重试。

进程资源回收机制¶

进程的退出 (Exit)¶

当进程调用 sys_exit 时，它将变成“僵尸 (Zombie)”状态，等待父进程回收。在本章的架构中，我们将这个逻辑放在 ProcManager 的 delete 之后的处理逻辑中。

但更准确地说，我们需要维护父子关系树。tg-task-manage 提供了基本的进程树支持。

 1// 伪代码逻辑：处理进程退出
 2
 3// 当 handle_syscall 返回 SyscallResult::Exit(code) 时
 4
 5let exit_code = code;
 6
 7// 1. 标记当前进程状态为 Zombie，记录 exit_code
 8// 在本章简化实现中，我们可能直接将 exit_code 写入某个共享位置，
 9// 或者利用 TaskControlBlock 的字段。
10
11// 2. 进程树维护：孤儿进程托管
12// 将当前进程的所有子进程挂载到 initproc 下
13// 这一步需要访问全局的进程树结构
14
15// 3. 资源回收
16// 物理页、页表等资源会随着 Process 结构的 Drop 自动回收。
17// 但为了让父进程能获取 exit_code，我们需要保留最小限度的信息（通常是 PID 和 code）。
18// 在本章简化版中，可以直接在 delete 时销毁 Process，
19// 但必须确保父进程能通过某种方式查到该子进程已退出。
20
21// 简化的处理逻辑：
22log::info!("proc {} exit", pid);
23PROCESSOR.get_mut().delete(pid);
24// 这里的 delete 会导致 Process Drop，回收所有资源。
25// 如果父进程此时调用 waitpid，它怎么知道子进程结束了？
26
27// 实际上，我们需要一个“僵尸队列”或者在 Process 中保留 Zombie 状态而不立即 delete。
28// 但 tg-task-manage 的设计可能倾向于立即回收。
29// 一种常见的做法是：sys_exit 不立即 delete，而是标记状态。
30// 真正的 delete 发生在父进程 waitpid 时。

父进程回收 (Waitpid)¶

sys_waitpid 的核心逻辑是查询子进程状态。

 1// 退出系统调用处理（在主循环中）
 2Id::EXIT => unsafe {
 3    (*processor).make_current_exited(ret)  // 标记当前进程退出
 4},
 5
 6// wait系统调用实现
 7fn wait(&self, _caller: Caller, pid: isize, exit_code_ptr: usize) -> isize {
 8    let processor: *mut PManager<ProcStruct, ProcManager> = PROCESSOR.get_mut() as *mut _;
 9    let current = unsafe { (*processor).current().unwrap() };
10
11    // 1. 查询僵尸子进程
12    if let Some((dead_pid, exit_code)) =
13        unsafe { (*processor).wait(ProcId::from_usize(pid as usize)) }
14    {
15        // 2. 将退出码写入用户空间
16        const WRITABLE: VmFlags<Sv39> = build_flags("W_V");
17        if let Some(mut ptr) = current.address_space
18            .translate::<i32>(VAddr::new(exit_code_ptr), WRITABLE)
19        {
20            unsafe { *ptr.as_mut() = exit_code as i32 };
21        }
22
23        // 3. 返回子进程PID（资源由PManager内部回收）
24        dead_pid.get_usize() as isize
25    } else {
26        // 4. 子进程不存在
27        -1
28    }
29}
